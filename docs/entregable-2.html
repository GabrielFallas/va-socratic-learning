1.2 Cambios Realizados y Justificación

Cambio 1 — Migración del Motor de Avatar y Pivote a Gamificación (corrección crítica + evolución)

Problema original: Ready Player Me cerró su servicio en enero de 2026, eliminando el pilar visual de la arquitectura original (avatares VRM 3D, lip-sync mediante visemas).

Solución adoptada (evolución final): Se migró hacia un avatar 2D gamificado implementado en Kaplay.js, usando el sprite de Sonic the Hedgehog (16 frames, 2 animaciones: run / jump) como representación visual del tutor socrático. El avatar opera mediante 8 estados expresivos reactivos (idle, thinking, speaking, listening, happy, curious, empathetic, encouraging) que se mapean a comportamientos de juego en tiempo real (Sonic corre, salta, se congela, activa modo victoria). El sistema incluye dos zonas temáticas que cambian por tarea: Chemical Plant (Tarea 1, fondo azul-verde) y Speed Highway (Tarea 2, tinte cálido naranja). Un sistema de anillos gamifica el progreso socrático: se gana un anillo cuando la respuesta del usuario se aproxima a la solución, y se pierde cuando el agente detecta frustración. Un mini-juego de transición Kaplay entre tareas refuerza la experiencia lúdica.

Justificación teórica: El Hajji et al. (2025) documentan que avatares 2D estilizados logran el mismo anclaje de atención pedagógica que los 3D con menor costo computacional. La estética cartoon de Sonic neutraliza el Valle Inquietante (Mori, 1970) y opera en el territorio de representación claramente artificial pero altamente expresiva. La gamificación con anillos añade un componente de motivación intrínseca (autodeterminación, Deci & Ryan, 2000) que el diseño CSS/SVG original no ofrecía.

Cambio 2 — Actualización de RQ1: "Perceived Pedagogical Support"

Problema: El término "fluidez pedagógica" no corresponde a ningún constructo validado en la literatura HCI ni educativa.

Solución adoptada: Se sustituyó por perceived pedagogical support, medido mediante una escala Likert ad hoc de 5 ítems adaptada de Essel et al. (2024). Este constructo está ampliamente documentado en la literatura de Sistemas de Tutoría Inteligente (ITS) y es directamente comparable entre condiciones experimentales.

Cambio 3 — Incorporación de la Condición de Control "Sin Avatar"

Problema: La ausencia de una condición baseline hacía imposible atribuir causalmente cualquier efecto observado al embodiment del agente.

Solución adoptada: Se diseñó e implementó la Condición B: un chat de texto plano que mantiene el mismo system prompt socrático, el mismo modelo de lenguaje y las mismas tareas, eliminando únicamente el avatar, la voz y los gestos. Esta comparación Condición A (avatar multimodal) vs. Condición B (texto plano) permite aislar el efecto causal del embodiment, respondiendo directamente a la brecha identificada en la literatura (Sonlu et al., 2024).

Cambio 4 — Conexión Explícita entre Andamiaje Pedagógico y Arquitectura Técnica

Problema: La teoría del andamiaje (Vygotsky, ZPD) no estaba vinculada formalmente con la lógica técnica del agente en el Entregable 1.

Solución adoptada: El system prompt socrático (sonic-system.ts) implementa un protocolo de escalado de pistas en tres niveles alineado con la Zona de Desarrollo Próximo: (1) preguntas exploratorias de alto nivel, (2) orientación conceptual, y (3) pista estructural específica sin proporcionar el código. El nivel de andamiaje aumenta únicamente si el participante lleva tres o más turnos sin avance. La justificación teórica HCI (Cassell et al., 2000; Bickmore & Picard, 2005; Yee & Bailenson, 2007) fue incorporada en el documento de protocolo de evaluación (docs/03-evaluation-protocol.md).

Cambio 5 — Migración del LLM a Infraestructura Local (Ollama + Gemma 3 12B)

Para garantizar la reproducibilidad de la PoC sin dependencias de API externas, el motor de razonamiento se migró de Google Gemini a Ollama corriendo Gemma 3 12B localmente (Q4_K_M, ~7.3 GB, GPU RTX 5070 Ti 16 GB). Esta decisión no afecta la calidad del diálogo socrático y permite ejecutar la PoC de forma autónoma sin claves de API.

Cambio 6 — Migración de TTS y STT a Motores Locales de Alta Calidad

La síntesis de voz migró de Web Speech API a Piper TTS neuronal (Docker local, español de alta calidad) con fallback automático a Web Speech API cuando el servidor Piper no está disponible. El reconocimiento de voz migró a Whisper STT local via ruta Next.js /api/stt. Ambas decisiones mejoran la calidad de la experiencia multimodal y mantienen la arquitectura completamente local sin dependencias de nube.

Cambio 7 — Ampliación del README con Instrucciones de Configuración

El README del repositorio fue actualizado con instrucciones detalladas para instalar Ollama, descargar el modelo Gemma 3, configurar el servidor Piper TTS con Docker, configurar variables de entorno y lanzar el servidor de desarrollo Next.js.

2. Estado Actual del Agente — Prueba de Concepto (PoC)

2.1 Descripción General de la PoC

La Prueba de Concepto corresponde a una aplicación web funcional completa construida con Next.js 14 (App Router) que implementa los dos brazos del diseño experimental: la Condición A — Sonic corporizado con canvas de juego Kaplay 2D, TTS Piper neuronal, STT Whisper, gamificación de anillos, zonas temáticas (Chemical Plant / Speed Highway), mini-juego de transición y SFX/BGM — y la Condición B — el mismo tutor socrático presentado como chat de texto plano sin elementos de embodiment. El sistema demuestra la integración real de todos los componentes principales de la arquitectura M-ITS propuesta.

3. Diseño Metodológico del Estudio

3.1 Tipo de Estudio

Se adoptará un diseño de evaluación piloto con expertos. Este tipo de estudio es el apropiado dado que:

  • La evaluación con usuarios reales (estudiantes de cursos activos) requiere autorización del Comité Ético Científico (CEC) de la UCR, trámite que excede el alcance temporal del proyecto de curso.
  • Los expertos en educación STEM o en sistemas de tutoría inteligente permiten obtener retroalimentación pedagógica y técnica de alta calidad con una muestra pequeña, suficiente para validar la viabilidad y usabilidad del prototipo (Nielsen & Landauer, 1993).
  • Los hallazgos del piloto informarán el diseño de un estudio controlado con usuarios reales en fases posteriores del proyecto.
Nota del curso: Para PF-3311 se trabajará con pilotos y/o evaluaciones con expertos, dado que la evaluación con usuarios reales requiere autorización del CEC.

3.2 Diseño Experimental

Se utiliza un diseño entre grupos (between-subjects) con dos condiciones:

CondiciónDescripciónComponentes activos
Condición A — Experimental
(Sonic Multimodal + Multimodal)		 Interacción con Sonic mediante canvas Kaplay 2D, TTS Piper neuronal (+ fallback Web Speech), STT Whisper, gamificación de anillos, zonas temáticas y SFX/BGM. Canvas Kaplay 2D (Sonic sprite) · TTS Piper + fallback Web Speech · STT Whisper · LLM socrático (Ollama Gemma 3 12B) · Anillos + Zonas + SFX

Al mantener constantes el modelo LLM, el system prompt socrático, las tareas asignadas y el ambiente de evaluación, cualquier diferencia significativa entre condiciones puede atribuirse causalmente a la presencia del embodiment multimodal, no a la calidad de la tutoría.

3.3 Participantes

Perfil de la Muestra

  • ¿Quiénes? Docentes, investigadores o profesionales senior con experiencia en educación STEM, diseño de interfaces conversacionales o sistemas de tutoría inteligente.
  • ¿Cuántos? Entre 4 y 8 participantes (2–4 por condición). Para un estudio piloto exploratorio, esta muestra es estándar en HCI y permite identificar los problemas de usabilidad y pertinencia pedagógica más relevantes.
  • Criterios de inclusión:
    • Experiencia demostrable en docencia o investigación en computación, ingeniería o educación STEM.
    • Conocimiento básico de Python suficiente para comprender el contexto de las tareas de depuración.
    • Acceso a un navegador web moderno (Chrome 90+, Edge 90+ o Safari 15+).
  • Criterios de exclusión:
    • Conocimiento previo del objetivo o hipótesis del estudio (para evitar el sesgo de demanda).
    • Haber interactuado previamente con la PoC en modo de prueba o desarrollo.

El reclutamiento se realizará mediante invitación directa dentro de la comunidad académica de la ECCI/UCR y los contactos del profesor del curso PF-3311. Los participantes recibirán una descripción breve y neutral del estudio por correo electrónico (sin revelar la hipótesis) y se coordinarán sesiones individuales de aproximadamente 45 minutos en un entorno sin distracciones. La participación es voluntaria y puede interrumpirse en cualquier momento.

6. Protocolos de Interacción

Se presentan dos escenarios de interacción detallados, diseñados para ser reproducibles y redactados de forma neutral, sin mencionar los términos "avatar", "embodiment" ni "presencia social", para evitar el sesgo de expectativa del participante (Efecto Proteus, Yee & Bailenson, 2007).

Escenario 1 — Depuración de Bucle Infinito (Principiante)

Contexto presentado al participante:
"Usted está trabajando con un estudiante de primer año de ingeniería. El estudiante le muestra el siguiente programa en Python que, según indica, debería imprimir los números del 1 al 5, pero nunca termina. Su tarea es interactuar con el sistema de apoyo disponible para identificar el error lógico en el código y ser capaz de explicarlo con sus propias palabras. No modifique el código; solo identifique el problema."

Código presentado al participante:

def print_numbers():
    counter = 1
    while counter <= 5:
        print(f"Número: {counter}")
        # ¿Por qué nunca termina?

print_numbers()

Tareas específicas a realizar:

  1. Observar el código durante 1–2 minutos sin interactuar con el sistema.
  2. Iniciar una conversación con el agente describiendo el comportamiento observado.
  3. Responder a las preguntas del agente y continuar el diálogo hasta identificar el error lógico.
  4. Al identificar el problema, presionar el botón "Resolví el problema" y describir verbalmente la solución.

Resultado esperado de la interacción:
El participante identifica que la variable counter nunca se incrementa dentro del bucle while, por lo que la condición counter <= 5 es siempre verdadera. La corrección consiste en añadir counter += 1 al final del cuerpo del bucle. El agente no menciona directamente el error; guía mediante preguntas sobre la condición de salida y el estado de la variable de control.

Ejemplo de flujo de diálogo esperado:

TurnoParticipanteSonic (ejemplo de respuesta socrática)
1"El programa no termina nunca.""Anomalía detectada en la ruta de ejecución. Analicemos la condición de salida: ¿qué valor debe alcanzar counter para que el bucle se detenga?"
2"counter debe ser mayor a 5.""Correcto. Ahora dime: ¿dónde exactamente, dentro del cuerpo del bucle, se actualiza el valor de counter?"
3"No veo que se actualice en ningún lado…""Estado del sistema: análisis correcto. Si counter nunca cambia, ¿qué implicación tiene eso para la condición de salida en cada iteración?"
4"La condición siempre será verdadera. ¡Ya entendí!""Excelente deducción autónoma. Has localizado el error de arquitectura. ¿Cómo reestructurarías el bucle para que avance correctamente?"

Duración estimada: 5–10 minutos (tiempo máximo: 10 minutos).

Criterio de éxito: El participante identifica verbalmente que counter no se incrementa, sin que el agente haya revelado directamente el error.

Escenario 2 — Análisis de Complejidad y Optimización Algorítmica (Intermedio)

Contexto presentado al participante:
"Usted está revisando el código de un estudiante de segundo año. El programa funciona correctamente —encuentra los números duplicados en una lista—, pero se vuelve extremadamente lento cuando la lista tiene miles de elementos. El estudiante no entiende por qué. Interactúe con el sistema de apoyo para identificar por qué el algoritmo es ineficiente y qué estructura de datos podría mejorar el rendimiento. No proponga código directamente; solo identifique el problema conceptual."

Código presentado al participante:

def find_duplicates(numbers):
    """Encuentra números duplicados en una lista."""
    duplicates = []
    for i in range(len(numbers)):
        for j in range(i + 1, len(numbers)):
            if numbers[i] == numbers[j]:
                if numbers[i] not in duplicates:
                    duplicates.append(numbers[i])
    return duplicates

# Prueba con lista grande
import random
big_list = [random.randint(1, 1000) for _ in range(10000)]
result = find_duplicates(big_list)
print(f"Duplicados encontrados: {len(result)}")

Tareas específicas a realizar:

  1. Analizar la estructura del algoritmo: identificar cuántos bucles existen y qué operaciones realiza cada uno.
  2. Interactuar con el agente para razonar sobre el concepto de complejidad temporal.
  3. Identificar cuántas operaciones realiza el algoritmo para una lista de N elementos.
  4. Proponer, en palabras, qué estructura de datos reduciría la complejidad y explicar por qué.

Resultado esperado de la interacción:
El participante identifica que el algoritmo tiene complejidad O(n³) en el peor caso: dos bucles anidados O(n²) más la búsqueda lineal not in duplicates O(n). La solución conceptual es utilizar un set (conjunto) para rastrear los elementos vistos, reduciendo la complejidad total a O(n). El agente guía mediante preguntas sobre qué hace cada bucle, cuántas veces se ejecuta y qué garantiza un conjunto frente a una lista en cuanto a tiempo de búsqueda.

6.1 Consideraciones Generales del Protocolo

  • Instrucciones neutras: Ninguna instrucción entregada al participante menciona "avatar", "embodiment" ni "presencia social", para evitar el sesgo de expectativa.
  • Asignación ciega a condición: El investigador carga la condición (A o B) sin revelar que existe una condición alternativa ni cuál es el objetivo de la comparación.
  • Tiempo máximo por tarea: 10 minutos. Si el participante no identifica el error en ese tiempo, la tarea se registra como no resuelta y se avanza a la siguiente.
  • Protocolo de des-escalada de frustración: Si el participante muestra signos de frustración (más de 3 turnos sin avance), Sonic escala automáticamente el andamiaje al siguiente nivel sin proporcionar el código de corrección.
  • Consentimiento informado: Todos los participantes firman un consentimiento informado antes de iniciar la sesión (código PF-3311, investigador: Gabriel Fallas López, unidad: ECCI/UCR). Los datos se vinculan únicamente a un ID alfanumérico anónimo.
